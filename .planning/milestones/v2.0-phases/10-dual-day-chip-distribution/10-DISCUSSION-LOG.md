## 单日期失败语义

### 问题：两个目标交易日的筹码分布状态应该如何判定？

| Option | Description | Selected |
|--------|-------------|----------|
| 完全独立 | 最新交易日和前一有效交易日各自有 `succeeded / failed / blocked / missing` 状态；一个失败不影响另一个可用。 | ✓ |
| 股票级整体状态 | 只要两个日期任一失败，整只股票的筹码分布都视为不可用。 | |
| 你决定 | 按现有架构选择最容易维护且不影响 Phase 11 展示的方案。 | |

**User's choice:** 完全独立。
**Notes:** 日期级独立是 Phase 10 缓存和重试的核心语义。

### 问题：如果一个日期成功、另一个日期失败，页面和后续 Phase 11 应该如何消费？

| Option | Description | Selected |
|--------|-------------|----------|
| 部分可用 | 成功日期照常展示/输出；失败日期显示独立失败状态和脱敏原因。 | ✓ |
| 后端可保存部分成功，但对外 API 暂时标记整只股票部分不可用 | Phase 11 再决定展示方式。 | |
| 你决定 | 以最少改动兼容现有结果快照和 Phase 11 计划。 | |

**User's choice:** 部分可用。
**Notes:** 成功日期不能被另一个日期失败拖累。

### 问题：如果某只入选股票在 60 日行情里只有 1 条有效日线，无法找到前一有效交易日，应该怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 最新日可处理，前一日标记 `missing` | 能保存最新日分布，同时明确前一日缺失原因。 | ✓ |
| 整只股票跳过筹码处理 | 避免半成品，但会丢掉最新日可用数据。 | |
| 视为数据异常失败 | 记录失败原因，等待下次刷新重试。 | |

**User's choice:** 最新日可处理，前一日标记 `missing`。
**Notes:** 缺前一日不是临时 provider 失败。

### 问题：目标日期来源以哪个数据为准？

| Option | Description | Selected |
|--------|-------------|----------|
| 以筛选用到的实际 K 线序列为准 | 最新有效交易日就是筛选结果 `latestTradeDate`，前一有效交易日来自同一只股票对应的已调整/有效 bar 序列，不按自然日推断。 | ✓ |
| 以市场级最近两个交易日为准 | 简单，但停牌/缺数据股票可能误判。 | |
| 以 `cyq_chips` 返回日期为准 | 容易变成事后匹配，不能可靠规划缓存任务。 | |

**User's choice:** 以筛选用到的实际 K 线序列为准。
**Notes:** 目标日期不能按自然日或市场级交易日推断。

## 缓存复用和重试

### 问题：已成功缓存的“股票 + 交易日”筹码分布，后续刷新时如何处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 成功即跳过 | 只要该股票该交易日已有完整分布，就不再请求 provider；除非以后有全量重建或维护命令。 | ✓ |
| 每次刷新都重新拉取最新结果 | 数据最“新”，但会浪费额度，也违背增量目标。 | |
| 按 TTL 过期重拉 | 更复杂，本阶段没有明确必要性。 | |

**User's choice:** 成功即跳过。
**Notes:** Phase 10 继续服务于增量刷新目标。

### 问题：`failed` 状态下次刷新如何处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 下次刷新自动重试 | `failed` 不视为完整缓存；后续刷新继续补这个股票-日期。 | ✓ |
| 只在手动“重试筹码”命令里重试 | 减少自动请求，但 Phase 10 还没有单独 UI/命令范围。 | |
| 保留失败状态，不自动重试 | 最省额度，但会长期卡住缺口。 | |

**User's choice:** 下次刷新自动重试。
**Notes:** `failed` 对应临时性错误。

### 问题：`blocked` 状态下次刷新如何处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 不自动重试 `blocked` | 保留脱敏原因，避免每次刷新重复打同一个确定性失败接口；只有目标日期变化或维护者清理状态后再请求。 | ✓ |
| 每次刷新都重试 `blocked` | 如果权限后来开通会自动恢复，但会反复消耗请求。 | |
| 只对 `empty_data` 重试，权限/配置类不重试 | 更精细，但状态规则略复杂。 | |

**User's choice:** 不自动重试 `blocked`。
**Notes:** `blocked` 和 `failed` 必须分开处理。

### 问题：怎样才算某个“股票 + 交易日”的筹码分布已完整缓存？

| Option | Description | Selected |
|--------|-------------|----------|
| 状态为 `succeeded` 且至少有 1 个价格档位 | 有完整替换事务、分布行非空，就可跳过后续请求。 | ✓ |
| 只要状态是 `succeeded` 就算完整 | 实现简单，但可能隐藏空分布写入问题。 | |
| 需要校验 percent 合计接近 100% | 更严格，但可能因 Tushare 口径/精度造成误判。 | |

**User's choice:** 状态为 `succeeded` 且至少有 1 个价格档位。
**Notes:** 空分布不能污染成功缓存。

## 旧筹码峰兼容

### 问题：Phase 10 引入完整筹码分布后，现有表格/K线仍依赖的旧“前三筹码峰”字段怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 从完整分布派生兼容字段 | 完整分布是唯一源数据；临时派生前三峰给现有结果表格和 K 线用，Phase 11 再删除旧展示。 | ✓ |
| Phase 10 直接移除旧筹码峰字段 | 会提前触碰 Phase 11 UI 范围，风险更大。 | |
| 保留旧 `chip_peak` 存储为独立源数据 | 会形成两个数据口径。 | |

**User's choice:** 从完整分布派生兼容字段。
**Notes:** 完整分布是唯一源数据。

### 问题：这个兼容派生应该覆盖哪些旧消费方？

| Option | Description | Selected |
|--------|-------------|----------|
| 覆盖现有全部旧消费方 | 结果快照、表格筹码峰列、K 线 overlay 继续可用；Phase 11 再统一移除/替换。 | ✓ |
| 只覆盖后端结果快照，不保证 UI 旧展示 | 实现少一点，但 Phase 10 可能导致当前页面退化。 | |
| 只写新完整分布，不做兼容 | 最纯粹，但会让 Phase 10 和 Phase 11 强耦合。 | |

**User's choice:** 覆盖现有全部旧消费方。
**Notes:** Phase 10 不应提前破坏当前页面。

### 问题：旧“筹码峰”兼容字段应该从哪个日期派生？

| Option | Description | Selected |
|--------|-------------|----------|
| 从最新有效交易日派生 | 等价替代当前 `latestTradeDate` 的旧行为，前一有效日完整分布只给 Phase 11 对比图使用。 | ✓ |
| 从两个日期中占比最高的峰派生 | 会混淆“当前筹码峰”的日期语义。 | |
| 不派生日期含义，只给价格 | 会丢失后续排查所需信息。 | |

**User's choice:** 从最新有效交易日派生。
**Notes:** 旧字段仍表示最新日筹码峰。

### 问题：如果最新有效交易日完整分布失败，但前一有效交易日成功，旧“筹码峰”兼容字段怎么显示？

| Option | Description | Selected |
|--------|-------------|----------|
| 旧字段按最新日状态显示失败/不可用 | 不拿前一日冒充当前筹码峰；前一日成功留给新分布接口/Phase 11。 | ✓ |
| 用前一有效日分布派生旧筹码峰 | 页面看起来更完整，但语义错误。 | |
| 隐藏旧字段 | 会提前改变当前 UI 行为。 | |

**User's choice:** 旧字段按最新日状态显示失败/不可用。
**Notes:** 不用前一日冒充当前筹码峰。

## 完整性判定

### 问题：如果一次日期区间请求只返回两个目标交易日中的一个，应该怎么落库？

| Option | Description | Selected |
|--------|-------------|----------|
| 按日期独立落库 | 返回的目标日保存为 `succeeded`；未返回的目标日标记为 `blocked/empty_data` 并记录脱敏原因。 | ✓ |
| 整只股票两天都失败 | 简单，但会浪费已返回日期。 | |
| 重试一次后再决定 | 可能增加请求，但不一定解决数据源本身缺日。 | |

**User's choice:** 按日期独立落库。
**Notes:** 保持日期级独立状态。

### 问题：如果区间请求返回了非目标日期的数据，应该怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 只保存目标两个日期，忽略非目标日期 | Phase 10 缓存边界清晰，避免把 provider 额外返回的数据混入当前需求。 | ✓ |
| 顺手保存所有返回日期 | 可能提高未来命中率，但扩展了本阶段范围。 | |
| 视为 provider 异常，整次失败 | 过严，可能浪费可用目标数据。 | |

**User's choice:** 只保存目标两个日期，忽略非目标日期。
**Notes:** 不扩大缓存范围。

### 问题：同一股票同一交易日的新分布写入时，如何避免旧价格档位残留？

| Option | Description | Selected |
|--------|-------------|----------|
| 单事务内先删该股票-日期旧档位，再插入新档位和状态 | 满足 Phase 10 成功标准，简单可验证。 | ✓ |
| 逐行 upsert | 代码看似简单，但删除消失的旧档位更容易漏。 | |
| 新建 run 快照，不替换旧数据 | 保留历史，但不符合“复用已成功数据”的当前缓存模型。 | |

**User's choice:** 单事务内先删该股票-日期旧档位，再插入新档位和状态。
**Notes:** 事务完整替换是强约束。

### 问题：如果目标日期返回了 0 个价格档位，应该是什么状态？

| Option | Description | Selected |
|--------|-------------|----------|
| `blocked/empty_data` | 不是完整缓存，不展示为成功；但也不当作临时网络失败无限自动重试。 | ✓ |
| `failed` | 下次自动重试；可能反复打空数据接口。 | |
| `succeeded` 但空数组 | 实现简单，但会污染完整性判定。 | |

**User's choice:** `blocked/empty_data`。
**Notes:** 空分布不是成功，也不是自动重试失败。
