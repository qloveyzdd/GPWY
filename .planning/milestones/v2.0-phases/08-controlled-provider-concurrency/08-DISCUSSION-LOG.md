## 并发额度划分

### 并发池结构

| Option | Description | Selected |
|--------|-------------|----------|
| 全局统一上限 | REST/tinyshare、行情和筹码共用一个并发池 | ✓ |
| 按任务类型分池 | 行情与筹码分别配置并发数 | |
| 全局上限 + 类型子上限 | 同时限制总并发和各类型占用 | |

### 默认并发数

| Option | Description | Selected |
|--------|-------------|----------|
| 2 | tinyshare 稳定性优先 | |
| 4 | 速度与稳定性折中 | |
| 8 | 更偏向速度 | ✓ |

### 连续限频

| Option | Description | Selected |
|--------|-------------|----------|
| 自动临时降并发 | 本次运行降低，成功后缓慢恢复 | ✓ |
| 始终保持上限 | 只依靠退避重试 | |
| 直接终止批次 | 连续限频后等待人工重试 | |

### 上限作用范围

| Option | Description | Selected |
|--------|-------------|----------|
| 整个服务进程共享 | 刷新、筹码和验证合计不超过上限 | ✓ |
| 仅限制刷新任务 | 验证不计入额度 | |
| 每个工作流独立 | 每个工作流分别最多 8 | |

**User's choice:** 一个服务进程级全局池，默认上限 8；限频时动态下降并缓慢恢复。
**Notes:** 所有 provider 请求均计入同一额度。

## 退避策略

### 尝试次数

| Option | Description | Selected |
|--------|-------------|----------|
| 3 次 | 首次请求加最多两次重试 | ✓ |
| 4 次 | 增加临时故障恢复机会 | |
| 5 次 | 成功率优先但可能显著拖慢 | |

### 等待节奏

| Option | Description | Selected |
|--------|-------------|----------|
| 指数退避加抖动 | 避免同步重试风暴 | ✓ |
| 固定间隔 | 行为简单可预测 | |
| 线性退避 | 等待增长较缓 | |

### 限频与网络错误

| Option | Description | Selected |
|--------|-------------|----------|
| 区分等待节奏 | 限频等待更久 | |
| 使用相同节奏 | 两类错误统一指数退避 | ✓ |
| 限频直接失败 | 不执行重试 | |

### 空数据与未知错误

| Option | Description | Selected |
|--------|-------------|----------|
| 都不重试 | 避免无效请求 | ✓ |
| 仅 unknown 重试 | 空数据不重试 | |
| 两者都重试 | 按临时错误处理 | |

**User's choice:** 最多三次尝试，指数退避加随机抖动；限频和网络错误使用相同节奏。
**Notes:** 只有限频触发降并发；配置、权限、空数据和未知错误均不重试。

## Worker 故障恢复

### 请求超时后的 Worker

| Option | Description | Selected |
|--------|-------------|----------|
| 终止并重建 | 不复用可能失控的 SDK 进程 | ✓ |
| 保留 Worker | 只让当前请求失败 | |
| 移除且不补充 | 本次运行降低池容量 | |

### 在途请求

| Option | Description | Selected |
|--------|-------------|----------|
| 重新入队并计入次数 | 使用统一三次尝试预算 | ✓ |
| 重新入队但不计次数 | Worker 故障不消耗请求预算 | |
| 直接失败 | 只重建 Worker 处理后续请求 | |

### 槽位重建预算

| Option | Description | Selected |
|--------|-------------|----------|
| 3 次 | 超过后禁用槽位 | ✓ |
| 1 次 | 快速失败 | |
| 5 次 | 更多恢复机会 | |

### 全部槽位失效

| Option | Description | Selected |
|--------|-------------|----------|
| 结束 tinyshare 批次 | 未完成请求明确失败并正常收尾 | ✓ |
| 单 Worker 继续 | 继续尝试但可能长期停滞 | |
| 自动切换 REST | 静默改变 provider | |

**User's choice:** 超时或崩溃后终止并重建，重新入队并消耗请求预算；每槽位最多重建三次。
**Notes:** 全部槽位失效后结束批次，不自动切换 REST。

## 请求调度顺序

### 基础优先级

| Option | Description | Selected |
|--------|-------------|----------|
| 验证 > 行情/复权 > 筹码 | 交互验证最高，核心行情其次 | ✓ |
| 严格 FIFO | 完全按入队时间 | |
| 工作流分批 | 当前批次结束后再接受其他类型 | |

### 防止饥饿

| Option | Description | Selected |
|--------|-------------|----------|
| 等待时间提升优先级 | 等待越久逐步升级 | ✓ |
| 固定配额轮转 | 按请求数量比例轮转 | |
| 不处理饥饿 | 始终使用固定类型优先级 | |

### daily 与 adj_factor

| Option | Description | Selected |
|--------|-------------|----------|
| 独立任务公平调度 | 失败和重试互不阻塞 | ✓ |
| 同日成对绑定 | 连续执行同一天两个请求 | |
| 分接口批量执行 | 全部 daily 后执行全部 factor | |

### 重试重新入队

| Option | Description | Selected |
|--------|-------------|----------|
| 原优先级并保留首次时间 | 兼顾类型优先级与防饥饿 | ✓ |
| 放到队首 | 尽快处理失败项 | |
| 队尾并重置等待时间 | 实现简单但可能饥饿 | |

**User's choice:** 验证最高、行情和复权其次、筹码最低，并使用等待时间提升优先级。
**Notes:** `daily` 与 `adj_factor` 独立排队；重试保留首次排队时间。
