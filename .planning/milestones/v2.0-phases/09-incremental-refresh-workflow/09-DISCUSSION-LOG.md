## 增量刷新与断点恢复边界

### 普通刷新已有 active cache 时，目标 60 个交易日应如何更新？

| Option | Description | Selected |
|--------|-------------|----------|
| 只补缺失 | 只请求目标 60 日内本地缺失或失败的日期；无新增且无缺失时不下载行情，只重新筛选。 | ✓ |
| 新代重建 | 每次普通刷新创建一个新 generation，但复用已有日期数据；隔离更强，但实现和验证更重。 | |
| 原地追加 | 直接在 active generation 上追加新增日期；最快，但必须非常小心避免半成品被筛选读取。 | |

**User's choice:** 只补缺失。
**Notes:** 普通刷新以减少重复请求和支持续跑为核心。

### 断点恢复的最小粒度怎么定？

| Option | Description | Selected |
|--------|-------------|----------|
| 按数据项恢复 | `daily` 和 `adj_factor` 分开记录状态；某交易日 `daily` 成功、`adj_factor` 失败时，下次只补 `adj_factor`。 | ✓ |
| 按交易日恢复 | 只要某交易日任一项失败，下次重拉该日的 `daily` 和 `adj_factor`。 | |
| 按刷新批次恢复 | 中断后只跳过整批已完成日期，失败日期整日重拉；不追求 item 级精细恢复。 | |

**User's choice:** 按数据项恢复。
**Notes:** 与 Phase 8 的独立调度模型一致。

### 当数据源没有新增交易日、且本地目标 60 日都完整时，普通刷新应做什么？

| Option | Description | Selected |
|--------|-------------|----------|
| 只重跑筛选 | 不重新下载行情/复权因子；更新股票状态可按需要执行，然后用现有缓存重跑筛选并发布最新筛选结果。 | ✓ |
| 完全跳过 | 不下载、不筛选，只返回“无新增数据”。 | |
| 仍重新校验行情 | 即使完整也重拉最近交易日做确认。 | |

**User's choice:** 只重跑筛选。
**Notes:** 明确满足“无新增不重复下载已有 60 日行情”。

### 如果本次增量刷新只完成了一部分行情数据就中断或失败，页面和筛选结果怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 保留旧结果，等待补齐 | 已成功的数据项写入缓存用于下次续跑，但只有目标 60 个交易日的 `daily` 和 `adj_factor` 都完整后，才运行并发布新筛选结果。 | ✓ |
| 用可用数据筛选 | 尽量用已经成功的数据运行筛选。 | |
| 回滚本次成功项 | 失败就删除本次写入。 | |

**User's choice:** 保留旧结果，等待补齐。
**Notes:** 不用不完整行情产生误判，同时保留断点续跑价值。

## 筛选结果发布时间点

### 刷新 job 的完成边界怎么定？

| Option | Description | Selected |
|--------|-------------|----------|
| 筛选完成即刷新成功 | 行情补齐并筛选成功后，刷新 job 标记成功，页面立即显示新筛选结果；筹码作为独立后台阶段继续运行，失败不影响刷新成功。 | ✓ |
| 筹码完成后才刷新成功 | 保持当前语义，所有阶段结束才算刷新成功。 | |
| 两段式成功 | 刷新 job 有 `market/screening succeeded` 和 `chip completed` 两个完成态。 | |

**User's choice:** 筛选完成即刷新成功。
**Notes:** 新筛选结果不再被筹码处理阻塞。

### 筹码后台阶段启动方式怎么定？

| Option | Description | Selected |
|--------|-------------|----------|
| 自动启动 | 每次新筛选结果发布后自动启动筹码后台处理；页面显示筹码仍在处理中或部分失败。 | ✓ |
| 用户手动启动 | 刷新只产出筛选结果，页面另给“获取筹码分布”按钮。 | |
| 运维命令启动 | 网页只显示筛选结果，筹码由命令行触发。 | |

**User's choice:** 自动启动。
**Notes:** 保持“手动刷新一次完成主流程”的体验。

### 如果筹码后台阶段失败，刷新结果和页面状态怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 保留筛选结果，显示筹码失败 | 筛选结果一直可用；页面显示筹码阶段失败数量/原因，后续刷新或阶段重试可继续补。 | ✓ |
| 标记整体刷新失败 | 筹码失败会让本次刷新显示失败。 | |
| 静默忽略 | 页面不强调筹码失败，只缺图时显示暂无数据。 | |

**User's choice:** 保留筛选结果，显示筹码失败。
**Notes:** 避免筹码接口问题掩盖已经可用的筛选结果。

### 如果用户在筹码后台阶段尚未完成时再次点击手动刷新，系统怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 阻止新刷新 | 只要任一后台刷新/筹码阶段仍在运行，就不启动新的手动刷新。 | ✓ |
| 允许新刷新并取消旧筹码 | 新刷新优先，旧筹码任务停止或结果丢弃。 | |
| 允许并行 | 新刷新和旧筹码并行运行。 | |

**User's choice:** 阻止新刷新。
**Notes:** 保护 provider 额度和缓存状态，避免任务叠加。

## 阶段进度与 UI 展示

### 阶段展示建议怎么分？

| Option | Description | Selected |
|--------|-------------|----------|
| 四阶段展示 | `股票列表` → `行情/复权` → `筛选` → `筹码处理`。 | ✓ |
| 三阶段展示 | `行情准备` → `筛选` → `筹码处理`。 | |
| 明细阶段展示 | `stock_basic L/P/D`、`trade_cal`、`daily`、`adj_factor`、`screening`、`chip` 都显示。 | |

**User's choice:** 四阶段展示。
**Notes:** 解释耗时和失败位置，同时避免 UI 过度运维化。

### 每个阶段显示哪些指标？

| Option | Description | Selected |
|--------|-------------|----------|
| 数量 + 失败 + 耗时 | 显示状态、总数、完成数、失败数、耗时；重试次数只在有失败或警告时显示。 | ✓ |
| 只显示状态和百分比 | 最简洁，但不利于后续制定性能验收标准。 | |
| 显示全部诊断指标 | 总数、完成、失败、重试、跳过、耗时、平均耗时、最后错误。 | |

**User's choice:** 数量 + 失败 + 耗时。
**Notes:** 保留性能基线所需指标，但控制页面噪声。

### 阶段失败信息应该显示到什么程度？

| Option | Description | Selected |
|--------|-------------|----------|
| 脱敏摘要 + 失败数量 | 页面显示阶段失败数量和脱敏错误摘要；不展示 token、路径、原始响应全文。 | ✓ |
| 只显示失败数量 | 最安全简洁，但排查问题要看日志/数据库。 | |
| 显示详细错误列表 | 页面上逐项显示失败股票/日期和错误。 | |

**User's choice:** 脱敏摘要 + 失败数量。
**Notes:** 平衡可排查性和安全性。

### 进度轮询和结果刷新怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 2 秒轮询，阶段完成后刷新结果 | 沿用现有 `/api/refresh/status` 轮询；当筛选阶段完成时刷新服务端结果快照，筹码阶段后续再刷新一次。 | ✓ |
| 只在整个后台任务结束后刷新 | 实现简单，但不能做到筛选完成后立即显示结果。 | |
| 改为实时推送 | 用 SSE/WebSocket 推送进度。 | |

**User's choice:** 2 秒轮询，阶段完成后刷新结果。
**Notes:** 复用现有 UI 轮询模式，不引入实时推送复杂度。

## 运维全量重建路径

### 全量重建应如何暴露？

| Option | Description | Selected |
|--------|-------------|----------|
| npm 脚本/CLI | 提供运维命令，只能在服务器 shell 中执行。 | ✓ |
| 隐藏 API | 提供受保护 API，但不在页面放按钮。 | |
| 管理页面按钮 | 网页提供二次确认按钮。 | |

**User's choice:** npm 脚本/CLI。
**Notes:** 明确满足“网页没有高成本重建入口”。

### 全量重建运行时，对旧结果怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 旧结果持续可用 | 全量重建写入新的 building generation；只有完整验证通过后才激活。失败时旧 active cache 和旧筛选结果继续可用。 | ✓ |
| 开始重建就隐藏旧结果 | 语义明确，但用户可能长时间没有可用结果。 | |
| 先删除再重建 | 最干净但风险最高。 | |

**User's choice:** 旧结果持续可用。
**Notes:** 全量重建可能按小时计算，旧结果不能被提前移除。

### 全量重建和普通刷新冲突时怎么处理？

| Option | Description | Selected |
|--------|-------------|----------|
| 互斥运行 | 任一普通刷新、筹码后台阶段或全量重建运行中，都拒绝启动另一个刷新类任务。 | ✓ |
| 全量重建优先 | 启动全量重建时取消或阻止其他任务。 | |
| 允许并行 | 运维效率高，但高风险。 | |

**User's choice:** 互斥运行。
**Notes:** 保护 provider 额度和 SQLite 状态。

### 全量重建命令完成后输出什么信息？

| Option | Description | Selected |
|--------|-------------|----------|
| 阶段摘要 + 耗时 + 失败摘要 | 命令行输出各阶段状态、总数、完成数、失败数、耗时和脱敏错误摘要。 | ✓ |
| 只输出成功/失败 | 最简单，但对性能基线和故障定位帮助不足。 | |
| 输出详细逐项日志 | 排查方便，但噪声大，且可能暴露 provider 细节。 | |

**User's choice:** 阶段摘要 + 耗时 + 失败摘要。
**Notes:** 为后续性能验收标准提供真实基线数据。
