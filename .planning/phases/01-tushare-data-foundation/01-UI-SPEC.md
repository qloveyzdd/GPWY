# Phase 01 - UI Design Contract

> Visual and interaction contract for Phase 1: Tushare Data Foundation. Generated by gsd-ui-phase and verified before planning.

## Scope

Phase 1 delivers a protected personal web entry and a data-source status page. It does not deliver the final screening table, stock detail chart, refresh cache workflow, or production deployment guide.

Primary user intent: confirm whether the server can safely read configuration and whether Tushare stock, market-price-basis, and chip-candidate interfaces are usable.

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn official planned during Phase 1 scaffold |
| Preset | Neutral light theme, CSS variables, 8px radius ceiling |
| Component library | shadcn components backed by Radix primitives |
| Icon library | lucide-react |
| Font | system sans-serif stack: Inter-compatible, fallback to system UI |

Implementation note: no `components.json` exists yet because the frontend has not been scaffolded. The Phase 1 planner must include shadcn initialization in the scaffold plan before UI components are implemented.

## Product Tone

The interface is an operational diagnostics console for one personal user. It must feel quiet, utilitarian, and scan-friendly.

Do not use marketing hero sections, decorative illustrations, gradient orbs, nested cards, oversized display type, or explanatory onboarding panels. The first screen after login should be the actual status workspace.

## Screen Inventory

| Screen | Purpose | Required Elements |
|--------|---------|-------------------|
| Login screen | Protect the entire app before any status or validation action is visible | Password field, "进入工作台" button, generic password error |
| Status workspace | Show last validation result and allow explicit re-validation | Header, status summary strip, validation sections, "重新验证数据源" button, last-run timestamp |
| Validation running state | Show that a manual validation is in progress without exposing raw API detail | Disabled primary button, compact progress text, section-level pending states |
| Failure state | Show sanitized cause and next step | Error category, affected interface, safe remediation copy |

## Layout Contract

| Area | Desktop Contract | Mobile Contract |
|------|------------------|-----------------|
| App frame | Full-width page background with constrained 1180px content width | Single column, 16px side padding |
| Header | 56px high, project name left, session action right | 52px high, project name left, icon action right |
| Status summary | Four equal tiles in one row: config, stock basic, price basis, chip candidate | Two columns if space permits, otherwise one column |
| Validation sections | One column of full-width section bands; no cards inside cards | One column |
| Primary action | Header right on desktop, below summary on mobile | Full-width button under summary |

The status workspace focal point is the summary strip plus the "重新验证数据源" button. Validation sections are secondary and should support scanning, not compete for attention.

## Visual Hierarchy

| Priority | Element | Treatment |
|----------|---------|-----------|
| 1 | Overall data-source health | First row summary strip with clear status labels and compact icons |
| 2 | Re-run validation action | Single accent primary button, disabled while running |
| 3 | Price-basis decision | Dedicated section because it affects MA20, MA60, and swing-high correctness |
| 4 | Chip candidate result | Dedicated section because unavailable chip data blocks later chip-peak work |
| 5 | Sanitized diagnostic detail | Muted text and small tables; never shown as raw JSON |

Icon-only actions must have accessible labels and tooltips. Prefer lucide icons for status and actions: `ShieldCheck`, `Database`, `Activity`, `LineChart`, `RefreshCw`, `AlertTriangle`, `CheckCircle2`, `CircleSlash`.

## Spacing Scale

Declared values must stay within this scale.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline metadata separators |
| sm | 8px | Compact control gaps, status label gaps |
| md | 16px | Default component padding and grid gaps |
| lg | 24px | Section padding, page header spacing |
| xl | 32px | Major workspace gaps |
| 2xl | 48px | Login form top spacing on desktop |
| 3xl | 64px | Maximum page-level spacing only |

Exceptions: 44px minimum hit target for interactive controls on touch devices; this is an accessibility exception, not a layout spacing token.

## Typography

Use exactly these four sizes and two weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 14px | 400 | 1.4 |
| Body | 16px | 400 | 1.5 |
| Heading | 20px | 600 | 1.25 |
| Display | 28px | 600 | 1.2 |

Rules:
- Use `Display` only for the login title or status workspace title.
- Use `Heading` for section titles.
- Use `Label` for status chips, metadata, table headers, and helper text.
- Letter spacing is 0.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #F8FAFC | Page background and quiet empty surfaces |
| Secondary (30%) | #FFFFFF | Section surfaces, summary tiles, forms |
| Accent (10%) | #2563EB | Primary validation button, active focus ring, selected status filter if introduced |
| Success | #15803D | Successful interface status only |
| Warning | #B45309 | Degraded price-basis fallback and permission-warning states |
| Destructive | #B91C1C | Authentication failure, token invalid, or blocked validation states |
| Border | #E2E8F0 | Section borders and table dividers |
| Text | #0F172A | Primary text |
| Muted Text | #64748B | Metadata and safe diagnostic summaries |

Accent reserved for: primary "重新验证数据源" button, keyboard focus ring, one active navigation or filter state if introduced later. Accent must not be used for every link, badge, icon, or status.

## Component Contract

| Component | Source | Contract |
|-----------|--------|----------|
| Button | shadcn official | Primary, secondary, ghost, and danger variants; primary used only for validation action |
| Input | shadcn official | Password input with server-validated error state |
| Alert | shadcn official | Sanitized blocking and warning states |
| Badge | shadcn official | Status categories: 未验证, 正常, 警告, 阻塞 |

## State Contract

| State | Visual Treatment | Required Content |
|-------|------------------|------------------|
| Not validated | Neutral badge, muted section text | What has not run and the primary next action |
| Running | Accent spinner icon, disabled primary button | Which validation group is currently running, without raw request data |
| Success | Success badge and concise sample fields | Interface name, field list, safe sample code/name, timestamp |
| Warning | Warning badge and amber alert | Fallback reason, impact, next step |
| Blocked | Destructive badge and red alert | Sanitized error category, affected interface, explicit blocked capability |

Status labels must be stable: `未验证`, `正常`, `警告`, `阻塞`.

## Data Display Contract

The status page may show:
- Token configured: yes/no only, never the token value.
- Tushare interface name and success/failure state.
- Field names returned by sample responses.
- Safe sample values such as stock code and stock name.
- Chosen price basis and reason.
- Error category and affected interface.

The status page must not show:
- Full raw Tushare payloads.
- `TUSHARE_TOKEN`, access password, cookies, headers, or request bodies.
- Stack traces, local file paths, server environment dumps, or full exception objects.

## Accessibility Contract

- All form controls must have labels.
- Password errors must be announced near the password field.
- Buttons must have 44px minimum hit target on touch devices.
- Status color must be paired with text and icon; color alone is not sufficient.
- Keyboard focus must be visible with the accent focus ring.
- The validation button must expose disabled and busy states to assistive technology.

## Responsive Contract

Desktop is the primary target. Mobile must remain usable, but Phase 1 does not require a fully optimized mobile workflow.

Minimum requirements:
- No horizontal scrolling below 390px viewport width.
- Summary tiles wrap before text truncates important state.
- Long endpoint names and error categories wrap inside their containers.
- Primary action remains reachable without opening hidden navigation.

## Registry Safety

| Registry | Blocks Used | Safety Gate |

No third-party shadcn registry is declared for Phase 1. Third-party registries are out of scope unless explicitly approved in a future planning step with source review evidence.
