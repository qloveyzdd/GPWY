# Phase 09 — UI Design Contract

> Visual and interaction contract for the incremental refresh workflow. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn |
| Preset | `radix-nova` (`npx shadcn info`: preset code `b0`, style `nova`, base `radix`, base color `neutral`, font `inter`) |
| Component library | Radix via shadcn components |
| Installed components | `alert`, `badge`, `button`, `input`, `separator`, `table`, `tooltip` |
| Icon library | `lucide-react` |
| Font | Inter fallback stack from `src/app/globals.css` |
| Styling | Tailwind CSS v4 with CSS variables in `src/app/globals.css` |

Source decisions:

- Use existing shadcn/Radix components and CSS variables; do not introduce a second component system.
- Use the existing page structure in `StatusWorkspace`: header actions, status cards, validation/refresh/result sections.
- Phase 9 UI scope is staged refresh visibility only; dual-day chip distribution charts remain out of scope.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, compact inline metadata separators |
| sm | 8px | Badge/icon gaps, compact control groups, table action gaps |
| md | 16px | Card padding on mobile, default section gaps, alert padding |
| lg | 24px | Card padding on desktop, header-bottom spacing, stage grid gaps |
| xl | 32px | Major section gaps when stage panel and results stack vertically |
| 2xl | 48px | Page-level separation for empty states or blocking state groups |
| 3xl | 64px | Reserved for full-page login/empty-state vertical rhythm only |

Exceptions:

- Interactive controls keep the existing minimum touch target of 44px (`min-h-11`) for accessibility. This is an interaction-size exception, not a spacing token.
- Existing chart/table internals may keep their established heights and column spacing; Phase 9 must not introduce new arbitrary spacing values.

## Typography

New Phase 9 UI text must use only these four sizes and two weights:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label / metadata | 14px | 400 | 1.4 |
| Body | 16px | 400 | 1.5 |
| Section heading | 20px | 600 | 1.25 |
| Page display | 28px | 600 | 1.2 |

Rules:

- Use weight `400` for descriptive text, counters, timestamps, and error detail.
- Use weight `600` for page title, section headings, stage names, and primary numeric values.
- Do not add new font sizes for retry count, duration, or failed count. Use 14px metadata with semantic color.
- Use tabular numbers for progress counts and durations.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f8fafc` / `background` | Page background and low-emphasis refresh workspace surface |
| Secondary (30%) | `#ffffff` / `card`, `#f1f5f9` / `muted` | Stage cards, status panels, table containers, inactive card backgrounds |
| Accent (10%) | `#2563eb` / `primary`, `accent`, `ring` | Primary CTA, active stage border/ring, focus-visible ring, active polling indicator |
| Destructive | `#b91c1c` / `destructive` | Failed stage count, failed badge, sanitized fatal error state only |
| Warning semantic | `#b45309` / existing chart warning color | Partial chip completion, retry warning, degraded-but-usable state only |

Accent reserved for:

1. Primary CTA: `开始增量刷新`.
2. Active stage card border/ring while a stage is running.
3. Focus-visible ring on interactive controls.
4. Active polling/refresh indicator icon.

Accent must not be used for every clickable element. Secondary/outline button styling remains the default for refresh-status and validation actions.

## Visual Hierarchy and Layout Contract

Primary screen focal point:

- While no refresh is running: header CTA `开始增量刷新` is the primary focal point.
- While refresh or chip processing is active: the four-stage progress panel is the primary focal point, with the active stage visually emphasized.
- After screening completes: results content remains readable immediately, while the chip stage card continues to show background progress.

Page layout:

1. Header: page title, one-line explanatory subtitle, primary refresh CTA, secondary status/validation actions.
2. Top status area: stage progress panel with four cards.
3. Supporting status sections: validation/config/cache summary can remain below the stage panel.
4. Results table/chart area: refresh after screening completion marker changes; do not wait for chip stage completion.

Stage cards:

| Stage | User-facing label | Primary metric | Secondary metadata |
|-------|-------------------|----------------|--------------------|
| Stock list | 股票列表 | completed / total stocks | duration, failed count if any |
| Market data | 行情/复权 | completed / total data items | daily/factor failures summarized, retry count only on warning/failure |
| Screening | 筛选 | matched count or completed / total evaluated stocks | duration, published marker |
| Chip | 筹码处理 | completed / total candidates | failed count, sanitized reason summary, background status |

Card states:

- `pending`: muted card, gray badge, no spinner.
- `running`: accent border/ring, pulse/spinner icon, counters update at polling cadence.
- `succeeded`: neutral card with success badge, duration visible.
- `partial`: warning badge, usable-result message visible.
- `failed`: destructive badge, failure count and sanitized summary visible.
- `skipped`: muted badge with explicit reason, used only when no candidates or no work is required.

Icon-only controls:

- Avoid new icon-only actions in Phase 9.
- If an icon-only control is unavoidable, it must have an accessible label and tooltip.

## Interaction Contract

Primary CTA:

- Label: `开始增量刷新`.
- Enabled only when no ordinary refresh, chip background stage, or full rebuild is active.
- Disabled copy or tooltip while blocked: `已有刷新任务正在运行，请等待当前阶段完成。`
- On click: start ordinary refresh through the existing web refresh endpoint; do not expose full rebuild.

Polling and page refresh:

- Poll `/api/refresh/status` approximately every 2 seconds while any refresh or chip stage is active.
- Trigger `router.refresh()` when the screening publication marker changes.
- Continue polling after refresh job success if chip stage remains `running`.
- Trigger a second `router.refresh()` when chip stage reaches a terminal state so chip status in results can update.

Progress display:

- Always show status, total, completed, failed, and duration for every known stage.
- Retry count appears only when retry count is greater than 0 and the stage is warning or failed.
- Error details must be sanitized summaries only; do not render token values, request headers, local paths, raw provider responses, or stack traces.

Full rebuild:

- No button, link, menu item, or web API affordance for full rebuild.
- If full rebuild is running from CLI, the web UI may show a blocking status: `全量重建正在运行，暂不能启动普通刷新。`

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | 开始增量刷新 |
| Busy CTA | 刷新进行中 |
| Blocked CTA tooltip | 已有刷新任务正在运行，请等待当前阶段完成。 |
| Empty state heading | 尚未生成筛选结果 |
| Empty state body | 先运行增量刷新；行情与筛选完成后，新的筛选结果会立即显示，筹码处理可继续在后台完成。 |
| Screening published message | 筛选结果已更新，筹码处理仍在后台继续。 |
| Chip partial message | 筛选结果可用；部分筹码处理失败，可在下一次刷新中继续补齐。 |
| Error state | 刷新未完成：{stageLabel} 失败 {failedCount} 项。请稍后重试普通刷新；旧筛选结果仍可查看。 |
| Sanitized reason label | 脱敏原因 |
| Duration label | 耗时 |
| Retry label | 重试 |
| Destructive confirmation | 无网页破坏性操作。全量重建仅通过服务器 CLI 执行，页面不提供确认入口。 |

Copy rules:

- Stage labels must use the locked Chinese names: `股票列表`、`行情/复权`、`筛选`、`筹码处理`.
- Avoid “成功/失败” alone as the entire message; include the affected stage and next safe action.
- Do not promise a fixed 5–10 分钟 refresh duration. Phase 9 must show measured duration and leave performance targets for later calibration.
