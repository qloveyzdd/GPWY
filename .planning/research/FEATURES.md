# 功能研究

**领域：** 增量市场数据刷新与双交易日筹码分布
**研究日期：** 2026-06-25
**置信度：** 高

## 必备能力

| 功能 | 必要性 | 复杂度 | 说明 |
|------|--------|--------|------|
| 增量交易日规划 | 不再重复下载已有日期 | 中 | 依据本地最大交易日和交易日历生成缺失日期 |
| 标准化行情缓存 | 增量更新的基础 | 高 | 原始日线和复权因子分别保存 |
| 受控并发 | 消除串行耗时但不冲击接口 | 中 | 并发上限必须可配置并有安全默认值 |
| 限频退避与有界重试 | 并行后必须保护稳定性 | 中 | 只重试可恢复错误，权限/配置错误立即终止对应项 |
| 断点恢复 | 防止长任务失败后从头开始 | 中 | 已写入日期和已缓存筹码日期自动跳过 |
| 持久 tinyshare worker | 消除每请求启动 Python 的固定成本 | 高 | worker 异常退出后可重建，不能无限重启 |
| 双交易日完整筹码分布 | 用户明确要求 | 中 | 最新有效交易日和前一有效交易日 |
| 双图独立状态 | 部分数据仍可使用 | 中 | 某一天失败时，另一天仍展示并说明原因 |
| 阶段进度与耗时 | 为后续性能标准建立数据 | 中 | 行情、筛选、筹码分别计数和计时 |
| 手动全量重建 | 增量状态损坏时可恢复 | 中 | 与普通刷新分离，明确高成本 |

## 建议保留的兼容能力

| 功能 | 价值 | 复杂度 | 建议 |
|------|------|--------|------|
| 表格中的筹码峰摘要 | 保持现有批量扫描效率 | 低 | 从完整分布动态派生最高占比价位，不再单独作为源数据保存 |
| 筛选完成后先发布结果 | 用户不必等待筹码全部完成 | 中 | 将行情/筛选成功与筹码 enrichment 状态解耦 |
| 可重试失败筹码项 | 避免重新跑完整行情刷新 | 中 | 提供同一筛选批次的失败项重试 |

## 明确不做

| 功能 | 暂不做原因 | 替代 |
|------|------------|------|
| 自动每日调度 | 本里程碑先验证新的刷新机制 | 继续手动触发 |
| 动态自动提高并发 | 容易在不同额度下产生不稳定行为 | 配置安全默认值，记录实测后再调整 |
| 历史全市场回填 | 不是解决当前刷新性能的必要条件 | 首次只引导建立筛选需要的 60 日窗口 |
| PostgreSQL / Redis / 外部队列 | 单实例个人工具没有必要承担运维复杂度 | SQLite + 进程内调度 |
| 自研筹码估算 | 不能保持数据口径 | 使用官方 `cyq_chips` |
| 筛选参数或算法修改 | 与性能和双图目标无关 | 保持 v1.0 规则不变 |

## 功能依赖

```text
标准化行情缓存
  └─> 增量交易日规划
       └─> 增量行情刷新
            └─> 原有筛选算法

受控请求调度器
  ├─> 增量行情并行获取
  └─> 筹码分布并行获取

完整筹码分布存储
  ├─> 筹码峰摘要派生
  └─> 最新日/前一日双图

阶段进度模型
  ├─> 页面进度展示
  └─> 后续性能基线
```

## 推荐的 v2.0 最小范围

- [ ] 新标准化数据表和兼容迁移路径。
- [ ] 普通刷新只获取缺失交易日。
- [ ] 行情和筹码请求共用受控调度器。
- [ ] tinyshare 使用持久 worker 池。
- [ ] 完整保存最新交易日和前一交易日筹码分布。
- [ ] 两个筹码分布图支持独立成功、失败和加载状态。
- [ ] 记录阶段进度、耗时、成功数、失败数和重试数。
- [ ] 手动全量重建可从空缓存恢复。

## 来源
